## §0. Резюме для утверждающего

**Что утверждаем:** единый стандарт квотирования значений переменных (правила R-1…R-12) и процесс его поддержания (M-1…M-5), по которому **таблица требований + контракт-тест являются источником правды**, а код обязан им соответствовать — не наоборот.

**Зачем:** за 9 лет — 47 issue и 5 разворотов поведения (#712 → #797 → #827 → #847 → #905), потому что поведение нигде не было записано, а тесты фиксировали «как получилось». Стандарт делает шестой разворот механически невозможным.

**Цена:** один BREAKING-класс (§4): голые `WHERE x = $var` с нечисловыми строками у falsy-переменных; миграция — одна строка. Всё остальное поведение не меняется (измерено: 16 клеток из 64).

## §2. Нормативные требования к поведению

Ключевые слова **ДОЛЖЕН / НЕ ДОЛЖЕН / МОЖЕТ** — в смысле RFC 2119. Колонка «Контракт» — ID кейсов в тесте. Каждое требование проверено живым кодом ветки (jest-прогон 2026-07-05: 64 клетки решётки + 7 позиционных проверок + 4 механизма).

| ID | Требование | Пример (вход → выход) | Контракт | Статус |
|---|---|---|---|---|
| **R-1** | **Дефолт.** Одиночное скалярное значение переменной, у которой `multi` и `includeAll` оба falsy, ДОЛЖНО интерполироваться **raw** в плоской позиции. | `= $v` → `= abc` | A1–A4 | 🔶 изменено (#905) |
| **R-2** | **Эквивалентность tri-state.** Значения `false`, `undefined`, `null` в `multi`/`includeAll` ДОЛЖНЫ быть неразличимы. Строгие сравнения (`=== false`) НЕ ДОЛЖНЫ использоваться. | `false ≡ undefined ≡ null` | A-EQ | 🔶 новое |
| **R-3** | **Truthy → quote.** При `multi: true` ИЛИ `includeAll: true` одиночная нечисловая строка ДОЛЖНА квотироваться и экранироваться. | `= $v` → `= 'abc'` | A5, A6 | ⚪ унаследовано |
| **R-4** | **Числа.** Числовые значения ДОЛЖНЫ оставаться raw при любом конфиге в плоских и IN-позициях. Исключения: repeated-путь (R-10 квотирует и числа, строка 6 матрицы) и причуда L-3. | `= $v` → `= 123` | A7, C4 | ⚪ унаследовано |
| **R-5** | **IN/tuple, скаляр.** Скаляр внутри `IN()` / `IN[]` / `NOT IN` / `GLOBAL IN` / `tuple()` ДОЛЖЕН квотироваться с экранированием **независимо от конфига переменной**. | `IN ($v)` → `IN ('abc')` | B1, B3, B4, B7 | ⚪ унаследовано (#847) |
| **R-6** | **IN, массив.** Массив в IN-контексте ДОЛЖЕН давать CSV в кавычках без добавления внешних скобок. | `IN ($v)` → `IN ('a','b')` | B2, B6, E2 | ⚪ унаследовано (#838) |
| **R-7** | **Массив вне IN.** Массив в прочих позициях ДОЛЖЕН давать литерал массива ClickHouse. | `hasAny($v,c)` → `['a', 'b']` | E1 | ⚪ унаследовано (#829) |
| **R-8** | **Конкатенация и литералы.** Переменная в конкатенации (`$a.$b`, `$v.suffix`) или внутри строкового литерала (`'pre$v'`) ДОЛЖНА быть raw независимо от конфига. | `FROM $db.$t` → `FROM d.t` | D1–D3 | ⚪ унаследовано (#797, #827) |
| **R-9** | **Приоритет IN.** При одновременном совпадении IN-контекста и конкатенации IN ДОЛЖЕН выигрывать. | `db.$v … IN ($v)` → `'abc'` | B5 | ⚪ унаследовано (#847) |
| **R-10** | **Repeated-панели.** Если интерполируемое значение ≠ `current.value` (repeat by variable), скаляр ДОЛЖЕН квотироваться — **включая числовые значения**. `$__all` ДОЛЖЕН разворачиваться в полный список до сравнения. Экранирование на этом пути не выполняется (L-4). | → `'mysql'`, `'123'` | C1–C5 | ⚪ унаследовано (#712) |
| **R-11** | **Ответственность за value-позиции.** В плоской позиции кавычки ставит автор SQL (`= '$var'`); плагин НЕ ДОЛЖЕН добавлять или дублировать их. Экранирование в плоской позиции стандартом **не гарантируется** (см. L-1). | `= '$v'` → `= 'abc'` | A8, D2 | 🔶 изменено (#905) |
| **R-12** | **Фиксированный конвейер.** Порядок приоритетов: 1 конкатенация → 1.5 идентификаторная позиция → 2 IN+массив → 3 IN+скаляр → 4 массив → 5 repeated → 6 falsy-guard → 7 escape. Новые правила ДОЛЖНЫ добавляться явной позицией в конвейере + строками контракта (процесс M-2). | — | все | 🔶 новое |
| **R-13** | **Идентификаторные позиции.** Скаляр сразу после `FROM`/`JOIN`/`INTO`/`TO`/`TABLE` ДОЛЖЕН интерполироваться raw независимо от конфига и repeated-флага (квотированный идентификатор никогда не валиден в ClickHouse — правило может только чинить). IN/tuple выигрывает у этой позиции (как в R-9). Ограничение детекторов (вердикт глобален на имя, комментарии не вырезаются) распространяется и сюда. | `FROM $v` → `FROM db.logs` при любом конфиге | G1–G8 | 🔶 новое (добавлено поверх #906) |

## §3.0. Входы решения (сигнатура)

Решение зависит **только** от следующих параметров. Сырые входы: **P1** `query` (текст SQL) · **P2** `value` · **P3** `variable.name` · **P4** `variable.multi` (только truthy/falsy) · **P5** `variable.includeAll` (только truthy/falsy) · **P6** `variable.options[]` (типы значений и порядок) · **P7** `variables[]` — снимок `templateSrv.getVariables()` (источник `current.value`).

Производные признаки-ветки: **D1** isInConcatenation(P1,P3) · **D2** needsComma/IN-контекст(P1,P3) · **D3** isArray(P2) · **D4** числовость(P2) · **D5** returnAsIs(P6 — первая «решающая» опция; причуда L-3) · **D6** isRepeated(P2,P7,P6) · **D7** содержимое строки — влияет только на текст экранирования.

**НЕ влияют:** `variable.type` (только косвенно через P4/P5), `label`/`hide`/`allValue`/`current` на самом объекте `variable`, прочие поля модели, настройки датасорса. **Обход нулевого уровня:** формат-спецификатор (`${var:raw}`, `${var:sqlstring}`) отключает функцию целиком — ни один параметр не участвует.

Тонкости: (а) `multi/includeAll/options` читаются из аргумента колбэка, а `current.value` — из P7 по имени; рассинхрон → isRepeated молча false; (б) в D5 решает **первая** опция типа number либо первая нечисловая строка — порядок опций значим.

## §3.1. Флоу и таблица истинности: параметры → шаг → результат

**Краткий флоу** (первое совпадение выигрывает):

```
$var встретился в SQL
│
├─ 0. ${var:формат}?                                    ─► Grafana сама, наш код не вызывается
├─ 1. конкатенация / '…литерал…' (и НЕ в IN)?           ─► raw
├─ 1.5 FROM/JOIN/INTO/TO/TABLE $v (и НЕ в IN)?           ─► raw   ← R-13
├─ 2. IN()/IN[]/tuple() + массив?                        ─► 'a','b'
├─ 3. IN()/IN[]/tuple() + скаляр?                        ─► строка → 'v'⁺   число → raw
├─ 4. массив (в любой другой позиции)?                   ─► ['a', 'b']
├─ 5. repeated (value ≠ current.value)?                  ─► 'v' — даже числа, БЕЗ экранирования (L-4)
├─ 6. multi и includeAll оба falsy?                      ─► raw   ← фикс #905
└─ 7. иначе (мультиселект/All)                           ─► строка → 'v'⁺   число → raw
                                     ⁺ с экранированием \ и '
```

Мнемоника: сначала **ГДЕ** стоит переменная (1–4), потом **ОСОБЫЙ ли случай** (5), и только в конце — **КАК настроена** (6–7). Плагин квотит только там, где сам уверен; в плоской позиции кавычки — ответственность автора SQL.

Машинный вид конвейера R-12 в осях D-параметров. Читать сверху вниз, **первое совпадение выигрывает**; ✔ = требуется «да», ✖ = требуется «нет», — = не проверяется. Изоморфна сценарной матрице §3 (колонка «§3» связывает строки).

| Шаг | D1 конкат | D2 IN | D3 массив | D6 repeated | Guard falsy(P4)∧falsy(P5) | D4∧D5 число | → Результат | Экранир. | §3 | Контракт |
|---|---|---|---|---|---|---|---|---|---|---|
| 1 | ✔ | ✖ | ✖ | — | — | — | raw | нет | 1 | D1–D3 |
| 1.5 | идент.-позиция ✔ | ✖ | ✖ | — | — | — | raw | нет | 1b | G1–G6 |
| 2 | — | ✔ | ✔ | — | — | поэлементно | `'a','b'` CSV | да* | 2 | B2, B6, E2 |
| 3 | — | ✔ | ✖ | — | — | ✔→raw / ✖→`'v'` | число raw / строка `'v'` | да | 3–4 | B1, B3–B5 |
| 4 | — | — | ✔ | — | — | поэлементно | `['a', 'b']` | да* | 5 | E1 |
| 5 | — | — | ✖ | ✔ | — | **игнорируется** | `'v'` — и числа | **нет** ⚠L-4 | 6 | C1, C4, C5 |
| 6 | — | — | ✖ | ✖ | ✔ | не доходит | raw | нет ⚠L-1 | 7 | A1–A4, A7, A8, C2 |
| 7 | — | — | ✖ | ✖ | ✖ | ✔→raw / ✖→`'v'` | число raw / строка `'v'` | да | 8–9 | A5, A6 |

\* поэлементное экранирование через `clickhouseEscape`.

Причинности, видимые в этом виде: ✖D2 на шаге 1 = правило «IN перебивает конкатенацию» (R-9); шаг 5 не проходит через `clickhouseEscape` → квотит числа и не экранирует (C4/C5, L-4); guard читается ровно один раз (шаг 6) — формальное доказательство «позиция перебивает конфиг»; D4∧D5 — общий под-предикат шагов 3 и 7.

**Карта потребления параметров:** D1 → только шаг 1 · D2 → шаги 1–3 · D3 → все шаги · D6 → только шаг 5 · Guard(P4,P5) → только шаг 6 · D4∧D5 → шаги 2,3,4,7 · D7 → только текст экранирования в 2,3,4,7. Модификаторы: `$__all` и пустой `current{}` вычисляют D6; порядок `options` вычисляет D5.

## §3. Матрица принятия решений (полное покрытие)

**Как читать:** сверху вниз, **выигрывает первая подходящая строка** (порядок = приоритеты конвейера R-12). «—» = входной параметр **не влияет** на результат в этой строке. Благодаря don't-care-столбцам 9 строк покрывают все комбинации входов без пропусков: (позиция × тип значения × isRepeated × конфиг) — любая комбинация попадает ровно в одну строку.

**Входы:**
- **Позиция** — где стоит `$v` в SQL: `конкат/литерал` (`$a.$b`, `$v.suffix`, `'pre$v'`), `идентификаторная` (`FROM/JOIN/INTO/TO/TABLE $v` — R-13), `IN/tuple` (`IN()`, `IN[]`, `NOT IN`, `GLOBAL IN`, `tuple()`), `плоская` (всё остальное: `= $v`, `SELECT $v`, `f($v)`…)
- **Значение** — `массив` / `число` (строка, проходящая `^[+-]?\d+(\.\d+)?$`) / `строка` (все прочие скаляры)
- **isRepeated** — интерполируемое значение ≠ `current.value` (repeat-by-variable панели)
- **Конфиг** — `falsy` = `multi` и `includeAll` оба ∈ {false, undefined, null} (R-2); `truthy` = хотя бы одно `true`

## §4. BREAKING-изменение и миграция

**Основной ломаемый класс:** голое `WHERE x = $var` с **нечисловой строкой** у falsy-переменной — на 3.4.x приходило `= 'val'`, теперь `= val`. Ошибка громкая (`Unknown identifier`); редкий тихий случай — совпадение значения с именем колонки. **Экзотическая грань (сценарий №15):** числовая строка у falsy-переменной со смешанными `options` — была `'123'`, станет `123`; против String-колонки возможен type mismatch.

**Миграция:** `WHERE x = $var` → `WHERE x = '$var'`. Отдельно — **когорта issue #809**: пользователям, убравшим кавычки по официальному совету 2025 года, необходимо их вернуть (запись в CHANGELOG есть).

**Не ломается:** числа, массивы, IN/tuple, конкатенация, литералы `'$var'`, repeated-панели — 48 из 64 измеренных клеток идентичны 3.4.x. Гарантия односторонности: строгое множество ⊂ нестрогого, поэтому изменение возможно только в направлении quoted → raw; ни одна клетка не может «начать квотиться».

**Попутно чинится:** скрытый крэш на 3.4.x — `null`-значение при undefined-конфиге падало с `TypeError` в `clickhouseEscape` (`null.replace`); falsy-guard возвращает значение до escape (измерено).

## §5. Известные ограничения (зафиксированы, не нормируются)

| ID | Ограничение | Контракт | План |
|---|---|---|---|
| **L-1** | Плоская позиция не экранирует кавычку в значении: `O'Brien` внутри авторских `'$var'` ломает SQL.
